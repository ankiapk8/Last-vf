Performance, Architecture & Code Quality

What & Why

Three key pages (deck-detail.tsx at 1,338 lines, decks.tsx at 1,438 lines, generate-form.tsx at 936 lines) have grown to sizes that slow down development and make bugs harder to spot. Image data is stored as base64 blobs directly in the Postgres database (via the image and sourceImage text columns on cards), making queries carry megabytes of data for any deck with visual cards. There is also no rate limiting on the expensive AI endpoints, making them trivially abusable. These issues compound over time.

Done looks like

Each of the three bloated files is split into focused sub-components/hooks with clear responsibilities; no single file exceeds ~400 lines
Visual card images (base64 JPEG thumbnails stored in cards.image and cards.sourceImage) are migrated to Replit Object Storage; the DB columns store a short URL/key instead of the raw base64
The /api/generate/stream and /api/explain endpoints have a simple sliding-window rate limiter (max 10 requests per IP per minute) that returns HTTP 429 with a clear message
The TanStack Query cache in App.tsx no longer persists large image blobs to localStorage (which could exceed the 5 MB storage quota for visual-heavy libraries), instead excluding image-heavy queries from the persisted cache
Out of scope

Full CDN integration for image delivery
Database query optimization beyond the image migration

4. Rate limiting on AI endpoints (artifacts/api-server/src/routes/generate.ts, explain.ts)

Add a simple in-memory sliding window rate limiter (src/lib/rate-limiter.ts) with a Map<string, number[]> keyed by IP:
export function createRateLimiter(maxRequests: number, windowMs: number) {
  const map = new Map<string, number[]>();
  return (ip: string): boolean => {
    const now = Date.now();
    const times = (map.get(ip) ?? []).filter(t => now - t < windowMs);
    if (times.length >= maxRequests) return false;
    times.push(now);
    map.set(ip, times);
    return true;
  };
}
Apply as Express middleware before the generate and explain route handlers: max 10 requests per IP per 60 seconds
Return { error: "Too many requests. Please wait a moment before generating again." } with HTTP 429
5. Exclude image blobs from localStorage cache persistence (App.tsx)

The shouldDehydrateQuery function currently caches any query whose key includes /decks, /cards, etc. Queries for deck cards include base64 image strings that can be megabytes each, filling localStorage quickly.
Add exclusion: when hydrating, strip image and sourceImage fields from card data before persisting:
dehydrateOptions: {
  shouldDehydrateQuery: (q) => { ... }, // existing check
  // Add a custom serializer that strips image blobs
}
Alternatively, set gcTime for card queries to 0 so they are never persisted (they re-fetch quickly):
// In useListDeckCards query options, add: gcTime: 0
The simplest fix: in shouldDehydrateQuery, also check that the query key does NOT include "DeckCards" or "listDeckCards".
