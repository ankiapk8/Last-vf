```
Build me a full-stack web study scheduler app for a final-year medical student.
Use React + Vite (TypeScript) for the frontend, Express (TypeScript) for the backend API,
PostgreSQL with Drizzle ORM for the database, and Replit Auth (OpenID Connect / PKCE)
for authentication. Use Tailwind CSS v4 + shadcn/ui components for styling.
Use TanStack Query (React Query) for all API calls.

─────────────────────────────────────────
AUTHENTICATION
─────────────────────────────────────────
- Use Replit Auth (OIDC with PKCE). Never show the word "Replit" in the UI.
- Backend routes: GET /api/login → redirect to OIDC provider,
  GET /api/callback → exchange code, upsert user to DB, create session,
  GET /api/logout → end session + redirect to OIDC end_session,
  GET /api/auth/user → return current user or null.
- Sessions stored in PostgreSQL `sessions` table (sid, sess jsonb, expire timestamp).
- Users stored in PostgreSQL `users` table (id varchar PK, email, first_name,
  last_name, profile_image_url, created_at, updated_at).
- Frontend: useAuth() hook that fetches /api/auth/user on mount.
  Returns { user, isLoading, isAuthenticated, login(), logout() }.
- AuthGate component: shows loading spinner, then a sign-in screen
  (with bullet-point feature list and a "Sign In to Continue" button)
  if not authenticated, otherwise renders children.
- WelcomeModal: shown once per user on first login (keyed by `welcomed-{userId}`
  in localStorage). Shows user avatar/initials, name, feature highlights, and
  an "Accept & Start Planning" button.
- User bar at top of home page: avatar or initials placeholder, name,
  "Final Year Medicine" subtitle, "Sign Out" ghost button.

─────────────────────────────────────────
DATA MODEL
─────────────────────────────────────────
Topic interface:
  id: string (random alphanumeric)
  name: string
  subject: string
  filesAndMedia: string (URL)
  videoLink: string (URL)
  universityLecturer: string
  amboss: string (URL)
  notes: string
  status: "Not Started" | "In Progress" | "Done" | "Revised"
  difficultyLevel: "Easy" | "Medium" | "Hard"
  priority: "Low" | "Medium" | "High"
  from: string

Topics stored in PostgreSQL `user_topics` table:
  user_id varchar NOT NULL → references users(id) ON DELETE CASCADE
  storage_key varchar NOT NULL
  topics jsonb NOT NULL DEFAULT '[]'
  updated_at timestamp with time zone
  PRIMARY KEY (user_id, storage_key)

API endpoints:
  GET  /api/topics          → returns { topics: Record<storageKey, Topic[]> } for authenticated user
  PUT  /api/topics/:storageKey → upsert topics array for that key; returns { topics }

─────────────────────────────────────────
SUBJECT HIERARCHY (14 leaf subjects)
─────────────────────────────────────────
Home page shows 5 parent subject cards:

1. Sub Medicine (🩺) — storageKeys: dermatology, family, emergency, forensic, radiology
   Sub-page shows 5 child cards navigating to individual topic pages.
2. Psychiatric (🧠) — storageKey: psychiatric
   Single leaf, navigates directly to topic page.
3. Sub Surgery (🔬) — storageKeys: ent, ophthalmology, orthopedic, neurosurgery, urology
   Sub-page shows 5 child cards.
4. Pediatric (👶) — storageKey: pediatric
   Single leaf, navigates directly.
5. Gynecology (🌸) — storageKeys: gynecology, obstetric
   Sub-page shows 2 child cards.

Route structure (wouter):
  /                        → Home
  /sub-medicine            → SubMedicine (5 child cards)
  /sub-medicine/dermatology → TopicPage (storageKey="dermatology")
  /sub-medicine/family      → TopicPage (storageKey="family")
  /sub-medicine/emergency   → TopicPage (storageKey="emergency")
  /sub-medicine/forensic    → TopicPage (storageKey="forensic")
  /sub-medicine/radiology   → TopicPage (storageKey="radiology")
  /psychiatric              → TopicPage (storageKey="psychiatric")
  /sub-surgery              → SubSurgery (5 child cards)
  /sub-surgery/ent          → TopicPage (storageKey="ent")
  /sub-surgery/ophthalmology → TopicPage
  /sub-surgery/orthopedic   → TopicPage
  /sub-surgery/neurosurgery  → TopicPage
  /sub-surgery/urology       → TopicPage
  /pediatric                → TopicPage (storageKey="pediatric")
  /gynecology               → Gynecology (2 child cards)
  /gynecology/gynecology    → TopicPage
  /gynecology/obstetric     → TopicPage

─────────────────────────────────────────
SCHEDULING ALGORITHM
─────────────────────────────────────────
- User picks a schedule start date (date picker, persisted to localStorage key "schedule-start-date").
  Default = today.
- Schedule window = remaining days in the start date's month (inclusive).
- Sort all topics across ALL 14 subjects combined:
  1. Family Medicine always first (subject order override)
  2. Then by priority: High=0, Medium=1, Low=2
  3. Then by groupIndex (subject order)
  4. Then alphabetically by topic name
- Spread N topics over W window days:
  - If N <= W: topic i gets day offset i (one topic per day, some days empty)
  - If N > W: topic i gets day offset floor(i/N * W) (multiple topics per day)
  - Clamp to window-1 maximum
- First Study Date = startDate + dayOffset
- Second Study Date = First Study Date + 14 days
- This is used for both the calendar display and the CSV export.

─────────────────────────────────────────
HOME PAGE FEATURES
─────────────────────────────────────────
Three collapsible accordion sections (CSS grid-template-rows animation, 300ms):

     Columns: Name, Subject, Parent Subject, Files and Media, Video Link,
     University Lecturer, Amboss, First Study Date, Second Study Date,
     Notes, Status, Difficulty Level, Priority, From
   - ZIP filename: "revision-schedule.zip"
   - CSV filename in ZIP: "{subject-slug}-{month-year}.csv"
   - All dates in YYYY-MM-DD format
   - Show column name pills at bottom as reference
   - Disable buttons when no topics exist

3. BACKUP & RESTORE (default: collapsed)
   Header: HardDriveDownload icon, title, subtitle, chevron.
   Content:
   - "Download Backup" button → exports all topics as JSON:
     { version: 1, exportedAt: ISO string, scheduleStartDate: string|null,
       topics: Record<storageKey, Topic[]> }
     Filename: "study-planner-backup-YYYY-MM-DD.json"
   - "Choose Backup File" button → restores from JSON, upserts all storageKeys

   - Show success (green + ShieldCheck icon) or error (red + TriangleAlert) status message

Below the 3 panels: animated calendar, then subject navigation cards.

─────────────────────────────────────────
ANIMATED CALENDAR VIEW

Full-month calendar grid showing the computed schedule.

Features:

  - Status toggle buttons (Not Started/In Progress/Done/Revised) → immediately saves
- Cells animate in on mount with staggered fade-in + zoom-in-95 per cell index
- Calendar is wrapped in a ref div for html2canvas capture

─────────────────────────────────────────
TOPIC MANAGER (per leaf subject page)
─────────────────────────────────────────
- Shows 3 stat cards: Total Topics, Not Started, High Priority
- "Add Topic" button → Dialog modal with full form
- "Export CSV" button → downloads subject-specific CSV using schedule dates

  Topic Name* (required), Status (select), Priority (select),

  Files and Media URL (text), Video Link URL (text),
  Amboss URL (text), From/Source (text), Notes (textarea)

  lecturer + from + clickable Files/Video/Amboss links, truncated notes

- Loading spinner while fetching from API

─────────────────────────────────────────
LOCALSTORAGE MIGRATION
```

### PROMPT 4 — Replit Auth backend

```
Implement Replit Auth (OpenID Connect + PKCE) in the Express API server.

Use the openid-client npm package. OIDC issuer URL: https://replit.com/oidc
Client ID = process.env.REPL_ID (set automatically by Replit).

Implement these routes in /api:
- GET /login: generate state/nonce/PKCE verifier, store in short-lived cookies,
  redirect to OIDC authorization URL with scope "openid email profile offline_access"
- GET /callback: validate state/nonce/PKCE, exchange code for tokens,
  upsert user to `users` table (id=sub, email, first_name, last_name, profile_image_url),
  create session (store user + tokens in `sessions` table, set httpOnly secure cookie),
  redirect to returnTo param
- GET /logout: delete session from DB, clear cookie, redirect to OIDC end_session URL
- GET /auth/user: return req.user if authenticated, else { user: null }
- POST /mobile-auth/token-exchange: exchange mobile auth code for session token
- POST /mobile-auth/logout: delete session

Sessions: store as JSON in `sessions` table. Session ID is a random string stored
in a "sid" httpOnly secure cookie. TTL = 7 days.

Add authMiddleware that reads the session cookie, looks up the session in DB,
attaches req.user if valid. Add req.isAuthenticated() method.

Use Zod schemas from @workspace/api-zod to validate and serialize responses.
```

### PROMPT 6 — Topic data types & scheduling logic

```
Create src/lib/topics.ts with all topic-related types and pure functions.

Types:
  Status = "Not Started" | "In Progress" | "Done" | "Revised"
  Difficulty = "Easy" | "Medium" | "Hard"
  Priority = "Low" | "Medium" | "High"
  Topic interface (id, name, subject, filesAndMedia, videoLink, universityLecturer,
    amboss, notes, status, difficultyLevel, priority, from)
  ScheduledItem interface (topic, subjectLabel, parentLabel, storageKey, firstDate, secondDate)
  SubjectGroup interface (parentLabel, subjectLabel, storageKey)

Schedule helpers:
  getScheduleStartDate() → reads localStorage "schedule-start-date" or returns today
  setScheduleStartDate(date) → writes to localStorage
  getScheduleWindowDays(startDate) → remaining days in that month (inclusive)
  isoDate(date) → "YYYY-MM-DD" string
  formatStartDate(date) → "Month D, YYYY" string

computeSchedule(groups, topicsMap, startDate?):
  1. Flatten all topics from all groups
  2. Sort: family first, then High>Medium>Low, then groupIndex, then name
  3. assignDates: if N<=W give each topic its own day; if N>W compress to fit W days
  4. Second study date = first + 14 days
  Returns ScheduledItem[]

CSV generation:
  generateSubjectCSV(topics, subjectLabel, parentLabel) → CSV string for one subject
  generateGroupCSV(groups, parentLabel, topicsMap) → combined CSV for a parent group
  generateAllSubjectsCSV(groups, topicsMap) → all 14 subjects combined
  generateSeparatedCSVs(groups, topicsMap) → SeparatedCSV[] (one per subject, with filename)
  downloadCSV(content, filename) → creates blob + anchor click
  downloadZip(files, zipName) → uses jszip, creates zip blob + anchor click

Backup:
  exportBackup(groups, topicsMap) → downloads JSON backup file
  importBackup(json) → parses and validates, returns { ok, message, data? }

ALL_SUBJECT_GROUPS: flat array of all 14 SubjectGroup entries

Color maps for UI badges:
  STATUS_COLORS, PRIORITY_COLORS, DIFFICULTY_COLORS (Record<T, Tailwind class string>)

NOTION_HEADERS: array of 14 column name strings (Name, Subject, Parent Subject,
  Files and Media, Video Link, University Lecturer, Amboss, First Study Date,
  Second Study Date, Notes, Status, Difficulty Level, Priority, From)
```
